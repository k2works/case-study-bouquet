# 非機能要件定義 - フレール・メモワール WEB ショップシステム

## 概要

本ドキュメントでは、フレール・メモワール WEB ショップシステムの非機能要件を定義する。各要件は測定可能な数値目標を含み、アーキテクチャ設計・テスト戦略・運用設計の基準として機能する。

### 対象システム

- **システム名**: フレール・メモワール WEB ショップシステム
- **アーキテクチャ**: レイヤード3層アーキテクチャ＋ドメインモデル
- **技術スタック**: Java 25 / Spring Boot / Thymeleaf（SSR）
- **永続化**: 単一 RDB（JPA）

### 推定の前提

ビジネスアーキテクチャおよびインセプションデッキから、以下のビジネス規模を推定する。

| 項目 | 推定値 | 根拠 |
|------|--------|------|
| 対象顧客 | 個人顧客（記念日に花を贈る層） | ビジネスモデルキャンバス: 顧客セグメント |
| チーム規模 | 開発者 1〜2 名 | インセプションデッキ: チーム構成 |
| 段階的リリース | フェーズ 1（MVP）→ 2 → 3 | インセプションデッキ: フェーズ分割 |
| 品質優先度 | 固定（最優先） | インセプションデッキ: トレードオフ |

## 1. 性能要件

### 1.1 レスポンスタイム

記念日に花を届けるサービスであり、顧客の注文体験が事業の要。WEB ショップとして快適な操作感を提供する。

| 操作 | p50 | p95 | p99 | 根拠 |
|------|-----|-----|-----|------|
| ページ表示（SSR） | 500ms 以内 | 1,000ms 以内 | 2,000ms 以内 | Thymeleaf SSR のサーバーサイドレンダリング |
| 商品一覧表示 | 300ms 以内 | 800ms 以内 | 1,500ms 以内 | 商品数は花束数十種類程度（データ量小） |
| 受注登録 | 500ms 以内 | 1,000ms 以内 | 2,000ms 以内 | トランザクション処理を含む |
| 在庫推移表示 | 500ms 以内 | 1,500ms 以内 | 3,000ms 以内 | 日別の在庫推移計算を含む |
| 管理画面一覧表示 | 300ms 以内 | 1,000ms 以内 | 2,000ms 以内 | ページネーション前提 |

### 1.2 スループット

| 指標 | 目標値 | 根拠 |
|------|--------|------|
| 同時注文処理数 | 10 件/秒 | 小規模個人向け WEB ショップ。ピーク時（母の日等）を想定 |
| バッチ処理（在庫推移計算） | 1 分以内 | 商品数・単品数が限定的（数十〜数百件） |

### 1.3 同時接続数

| 指標 | 目標値 | 根拠 |
|------|--------|------|
| 通常時同時接続ユーザー数 | 50 ユーザー | 個人顧客向け WEB ショップの通常負荷 |
| ピーク時同時接続ユーザー数 | 200 ユーザー | 記念日シーズン（母の日・クリスマス等）を想定 |
| 管理画面同時利用者数 | 5 ユーザー | スタッフ数（受注・仕入・フローリスト） |

## 2. セキュリティ要件

### 2.1 認証・認可

顧客情報（届け先住所・メッセージ）と決済関連情報を保護する。

| 要件 | 仕様 | 根拠 |
|------|------|------|
| 認証方式 | Spring Security によるフォームベース認証 | SSR アーキテクチャとの親和性。技術スタック: Spring Boot |
| パスワードポリシー | 最低 8 文字、英数記号混在 | 個人情報保護の基本要件 |
| パスワード保存 | bcrypt ハッシュ化 | Spring Security 標準 |
| セッション管理 | サーバーサイドセッション、30 分タイムアウト | SSR アプリケーションの標準 |
| 認可モデル | RBAC（Role-Based Access Control） | 得意先・スタッフの 2 ロール。シンプルなロール構成 |
| ロール定義 | `CUSTOMER`（得意先）、`STAFF`（管理者） | システムコンテキスト: 得意先とスタッフの 2 アクター |

### 2.2 データ暗号化

| 要件 | 仕様 | 根拠 |
|------|------|------|
| 通信暗号化 | TLS 1.2 以上（HTTPS 必須） | 個人情報（住所・電話番号）の通信経路保護 |
| パスワード | bcrypt ハッシュ化（不可逆） | 業界標準 |
| DB 接続 | SSL/TLS 接続 | DB に保存される個人情報の経路保護 |
| 保存データ暗号化 | RDB のストレージ暗号化（透過的暗号化） | 個人情報保護法への準拠 |

### 2.3 監査ログ

| 要件 | 仕様 | 根拠 |
|------|------|------|
| ログイン・ログアウト | 日時・ユーザー ID・結果を記録 | アクセス追跡 |
| データ変更操作 | 受注・発注・出荷の作成・更新・削除を記録 | ビジネスプリンシプル: 届け日厳守。変更追跡が必要 |
| 管理操作 | マスタデータの変更を記録 | 商品・単品・仕入先の変更履歴 |
| ログ保持期間 | 1 年間 | 法令準拠および事業運営上の要件 |

## 3. 可用性要件

### 3.1 稼働率目標（SLA）

| 指標 | 目標値 | 根拠 |
|------|--------|------|
| 稼働率 | 99.5%（月間許容ダウンタイム: 約 3.6 時間） | 小規模 WEB ショップ。記念日需要は重要だが、24/365 の超高可用性は不要 |
| 計画停止 | 月 1 回、深夜帯（2:00-5:00）に最大 3 時間 | メンテナンスウィンドウの確保 |
| 計画停止の事前通知 | 3 営業日前 | ステークホルダーへの影響最小化 |

### 3.2 障害復旧時間（RTO）

| 障害レベル | RTO | 対応 |
|-----------|-----|------|
| 軽微（一部機能の不具合） | 4 時間以内 | パッチ適用またはロールバック |
| 重大（サービス全停止） | 8 時間以内 | システム復旧・データ復元 |
| 壊滅的（データ損失を伴う障害） | 24 時間以内 | バックアップからの完全復旧 |

### 3.3 データ復旧時点（RPO）

| 指標 | 目標値 | 根拠 |
|------|--------|------|
| RPO | 1 時間以内 | 受注データの損失は顧客体験に直結。1 時間分の受注データ損失を許容上限とする |
| バックアップ頻度 | 日次フルバックアップ + 1 時間ごとのトランザクションログバックアップ | RPO 1 時間を満たす設計 |

## 4. 保守性要件

### 4.1 ログ出力

| 要件 | 仕様 | 根拠 |
|------|------|------|
| ログレベル | ERROR / WARN / INFO / DEBUG の 4 段階 | Spring Boot 標準（SLF4J + Logback） |
| 本番ログレベル | INFO 以上を出力 | 必要な運用情報を取得しつつ、ディスク消費を抑制 |
| ログ出力先 | ファイル出力（日次ローテーション） | 単一サーバー構成。集約ログ基盤は将来検討 |
| ログ保持期間 | 30 日間（アプリケーションログ）、1 年間（監査ログ） | 運用上の問題調査期間と法令準拠 |
| ログフォーマット | 構造化ログ（JSON 形式） | 将来のログ集約基盤への対応を考慮 |

### 4.2 監視項目

| 監視対象 | メトリクス | 閾値（Warning） | 閾値（Critical） |
|---------|-----------|----------------|-----------------|
| CPU 使用率 | 平均 CPU 使用率 | 70% 超過（5 分間平均） | 90% 超過（5 分間平均） |
| メモリ使用率 | JVM ヒープ使用率 | 70% 超過 | 85% 超過 |
| ディスク使用率 | ディスク使用率 | 70% 超過 | 85% 超過 |
| レスポンスタイム | p95 レスポンスタイム | 1,500ms 超過 | 3,000ms 超過 |
| エラー率 | HTTP 5xx の割合 | 1% 超過（5 分間平均） | 5% 超過（5 分間平均） |
| DB コネクション | アクティブコネクション数 | プールの 70% 超過 | プールの 90% 超過 |

### 4.3 アラート設定

| 重要度 | 通知先 | 対応 |
|--------|--------|------|
| Critical | 開発者（即時通知: メール + チャット） | 即座に対応開始。30 分以内に初動報告 |
| Warning | 開発者（メール通知） | 翌営業日までに確認・対応 |
| Info | ログ記録のみ | 定期レビューで確認 |

## 5. 拡張性要件

### 5.1 スケーラビリティ

| 要件 | 方針 | 根拠 |
|------|------|------|
| 初期構成 | 単一サーバー構成 | 小規模チーム（1〜2 名）・個人顧客向け。過剰なインフラ投資を避ける |
| 垂直スケーリング | サーバースペックの増強で対応 | 単一 RDB 構成のため、まず垂直スケーリングで対応 |
| 水平スケーリング | 将来検討（フェーズ 3 以降） | ステートレスな Spring Boot アプリケーションは水平スケーリングに対応可能な設計を維持 |
| DB スケーリング | リードレプリカの追加（将来検討） | 在庫推移表示の参照負荷が高い場合に備える |

### 5.2 将来の拡張シナリオ

| シナリオ | 対応方針 | 影響範囲 |
|---------|---------|---------|
| 商品数の増加（数百種類） | ページネーション・キャッシュの導入 | プレゼンテーション層・アプリケーション層 |
| 顧客数の増加（数千人規模） | DB インデックス最適化・コネクションプール調整 | インフラ層 |
| 法人顧客対応 | RBAC のロール追加・マルチテナント検討 | ドメインモデル・認可設計 |
| モバイルアプリ対応 | REST API の追加（現在の SSR と並行） | プレゼンテーション層の追加 |
| 外部サービス連携（決済・配送） | API ゲートウェイの導入検討 | インフラ層 |

## 非機能要件のトレーサビリティ

| 非機能要件 | 関連するビジネスプリンシプル | 関連するアーキテクチャ要素 |
|-----------|--------------------------|-------------------------|
| レスポンスタイム | WEB ファースト | Thymeleaf SSR、Spring MVC |
| セキュリティ（認証・認可） | リピーター重視 | Spring Security、RBAC |
| 監査ログ | 届け日厳守 | SLF4J + Logback |
| 稼働率 99.5% | 鮮度ファースト | 単一サーバー構成 |
| RPO 1 時間以内 | 届け日厳守 | DB バックアップ戦略 |
| 拡張性（段階的スケーリング） | シンプルな業務フロー | レイヤード3層アーキテクチャ |
