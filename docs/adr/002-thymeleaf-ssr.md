# ADR-002: SSR（Thymeleaf）を採用しフロントエンド SPA を不採用

日付: 2026-03-30

## ステータス

承認済み

## コンテキスト

フレール・メモワール WEB ショップシステムのフロントエンドアーキテクチャを決定する必要がある。本システムは受注管理・在庫管理・発注管理を中心とした業務システムであり、顧客向け 8 画面 + 管理向け 7 画面 = 計 15 画面で構成される。

検討にあたり、以下の制約と要件がある:

- 開発チームは 1〜2 名の小規模構成
- バックエンドフレームワークとして Spring Boot を採用済み（ADR-001）
- フォームベース認証・セッション管理を採用（非機能要件: セキュリティ）
- SEO 要件は限定的（管理画面が主体）
- リアルタイム通信やリッチな UI インタラクションの要件はない

## 決定

フロントエンドアーキテクチャとして **Thymeleaf による SSR（サーバーサイドレンダリング）** を採用し、SPA（Single Page Application）フレームワークは不採用とする。

採用理由:

1. **アーキテクチャの単純化**: フロントエンド/バックエンドの分離が不要となり、1つの Spring Boot アプリケーションで完結する
2. **開発効率**: テンプレートエンジンにより、サーバーサイドで完結する画面開発が可能。ビルドパイプラインの複雑化を回避できる
3. **Spring Boot との統合**: Thymeleaf は Spring Boot の公式テンプレートエンジンであり、Spring Security（CSRF トークン、認証情報）との統合が標準で提供される
4. **テスト容易性**: MockMvc による Controller + テンプレートの統合テストが可能で、TDD サイクルに組み込みやすい
5. **小規模チームへの適合**: フロントエンド専門スキル（React/Vue の状態管理、ビルドツール設定等）が不要

## 検討した選択肢

### React + Next.js（SPA/SSR ハイブリッド）

- **利点**: リッチな UI、コンポーネントベース開発、大規模エコシステム
- **不採用理由**: フロントエンド/バックエンドの分離により開発・デプロイの複雑性が増す。API 設計（BFF or REST）が追加で必要。1〜2 名のチームでフロントエンド・バックエンド両方の技術スタックを維持するコストが高い

### Vue.js + Nuxt

- **利点**: 学習コストが React より低い、SSR サポート
- **不採用理由**: React と同様、フロントエンド/バックエンドの分離に伴う複雑性が課題。本システムの画面要件（フォーム入力中心の業務画面）に対してオーバースペック

### JSP（JavaServer Pages）

- **利点**: Java EE 標準、長い実績
- **不採用理由**: Spring Boot での推奨度が低い。Thymeleaf と比較してテンプレートの可読性が劣る。ナチュラルテンプレーティング（HTML として直接開ける）が使えない

## 影響

### 肯定的な影響

- フロントエンドのビルドプロセスが不要となり、開発・デプロイパイプラインが単純化される
- Spring Security との統合がシームレスであり、認証・認可の実装コストが低い
- サーバーサイドで全ての状態管理が完結するため、状態管理の複雑性が排除される

### 否定的な影響

- ページ遷移のたびにサーバーへのリクエストが発生し、SPA と比較して画面遷移の体感速度が劣る場合がある（htmx による部分更新で緩和可能）
- リッチな UI インタラクション（ドラッグ&ドロップ、リアルタイム更新等）の実装が SPA より困難
- フロントエンドエンジニアの採用時に Thymeleaf の経験者が少ない可能性がある

## コンプライアンス

- 全画面が Thymeleaf テンプレート（`.html`）で実装されていることを確認する
- JavaScript フレームワーク（React, Vue 等）の依存が追加されていないことを `build.gradle` で確認する
