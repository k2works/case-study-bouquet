# ADR-004: 永続化層に MyBatis（XML マッパー方式）を採用

日付: 2026-03-30

## ステータス

承認済み

## コンテキスト

フレール・メモワール WEB ショップシステムの永続化層の実装技術を決定する必要がある。技術スタック選定（`docs/design/tech_stack.md`）では当初 Spring Data JPA が記載されていたが、実装開始にあたり永続化技術の再検討が求められた。

本システムのデータモデルは 10 エンティティ（得意先、届け先、受注、商品、商品構成、単品、仕入先、発注、入荷、在庫）で構成され、以下の特性がある:

- テーブル間の外部キー制約が多く、JOIN を含む複雑なクエリが必要になる場面がある
- ビジネスルール（BR01-BR07）に基づく在庫推移計算等、SQL レベルでの最適化が必要になる可能性がある
- 開発チームは SQL を明示的に管理し、発行される SQL を把握したい方針がある

## 決定

永続化層に **MyBatis（XML マッパー方式）** を採用する。Spring Data JPA は使用しない。

採用理由:

1. **SQL の明示的管理**: XML マッパーファイルに SQL を直接記述するため、発行される SQL が一目で把握できる
2. **複雑なクエリへの対応**: JOIN、サブクエリ、動的 SQL を XML で柔軟に記述できる
3. **学習コストの予測可能性**: SQL の知識がそのまま活用でき、ORM 固有の概念（遅延ロード、N+1 問題、キャッシュ戦略等）の学習が不要
4. **Spring Boot との統合**: `mybatis-spring-boot-starter` により、Spring Boot の自動構成機能を活用できる
5. **ドメインモデルとの分離**: MyBatis はドメインモデルクラスに ORM アノテーションを要求しないため、ドメインモデルの技術独立性を維持できる

## 検討した選択肢

### Spring Data JPA

- **利点**: Repository インターフェースの自動実装、JPQL によるクエリ、Spring Boot との深い統合
- **不採用理由**: ORM の自動生成 SQL は予期しないクエリが発行される可能性がある。ドメインモデルクラスに `@Entity` 等の JPA アノテーションが必要となり、ドメインモデルの技術独立性が損なわれる

### MyBatis（アノテーション方式）

- **利点**: Java コード内で SQL を完結でき、XML ファイルが不要
- **不採用理由**: 複雑な SQL や動的 SQL の記述がアノテーション内では可読性が低下する。XML 方式の方が SQL とマッピングの分離が明確

### jOOQ

- **利点**: タイプセーフな SQL ビルダー、コンパイル時のスキーマ検証
- **不採用理由**: 有償ライセンスが必要（PostgreSQL 用）。ビルド時のコード生成ステップが必要で、開発ワークフローが複雑化する

## 影響

### 肯定的な影響

- SQL が XML ファイルに集約され、クエリの棚卸しやパフォーマンスチューニングが容易になる
- ドメインモデルクラスが POJO のまま維持され、フレームワーク依存がなくなる
- Flyway マイグレーションと MyBatis マッパーの SQL が同じ SQL 方言で記述され、一貫性が保たれる

### 否定的な影響

- CRUD 操作のボイラープレート SQL を各エンティティ分手動で記述する必要がある
- Spring Data JPA の自動実装（`findByXxx` メソッド等）が利用できず、全クエリを手動で定義する必要がある
- XML マッパーファイルの結果マッピングが手動のため、スキーマ変更時の追従作業が増える

## コンプライアンス

- `build.gradle` に `mybatis-spring-boot-starter` が含まれ、Spring Data JPA 関連の依存が含まれていないことを確認する
- ドメインモデルクラスに MyBatis 固有のアノテーションが含まれていないことを確認する
- 全 Mapper インターフェースに `@Mapper` アノテーションが付与されていることを確認する
- XML マッパーファイルが `src/main/resources/mapper/` 配下に配置されていることを確認する
