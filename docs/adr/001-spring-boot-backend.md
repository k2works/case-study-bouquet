# ADR-001: バックエンドフレームワークとして Spring Boot を採用

日付: 2026-03-30

## ステータス

承認済み

## コンテキスト

フレール・メモワール WEB ショップシステムのバックエンドフレームワークを選定する必要がある。本システムはレイヤード3層アーキテクチャ＋ドメインモデルパターンを採用し、受注管理・在庫推移・発注管理を中核とする業務システムである。開発チームは 1〜2 名の小規模構成であり、学習コストと開発効率のバランスが重要となる。

要件として以下がある:

- Java 25 を実装言語として採用済み
- SSR（サーバーサイドレンダリング）による Web アプリケーション
- 単一 RDB（PostgreSQL）への永続化
- Spring Security によるフォームベース認証・RBAC
- TDD を前提とした開発プロセス

## 決定

バックエンドフレームワークとして **Spring Boot 3.x** を採用する。

採用理由:

1. **自動構成（Auto-configuration）**: 最小限の設定でアプリケーションを起動でき、小規模チームの開発効率を最大化する
2. **Spring エコシステムの統合**: Spring MVC、Spring Security、Spring Data JPA、Thymeleaf との統合が標準で提供される
3. **豊富なドキュメントとコミュニティ**: 日本語ドキュメントを含む広範な情報源があり、問題解決が容易
4. **テスト支援**: Spring Boot Test、MockMvc、Testcontainers との統合により、TDD サイクルを効率的に実践できる
5. **運用支援**: Spring Boot Actuator による監視エンドポイント、プロファイル管理による環境切り替えが標準装備

## 検討した選択肢

### Quarkus

- **利点**: GraalVM ネイティブコンパイルによる高速起動、低メモリ消費
- **不採用理由**: Spring エコシステムとの互換性が限定的。Thymeleaf の統合が Spring Boot ほど成熟していない。コミュニティ規模が Spring Boot に比べ小さく、小規模チームでの問題解決コストが高い

### Micronaut

- **利点**: コンパイル時 DI による高速起動、低メモリ
- **不採用理由**: Spring Boot と比較してエコシステムが小さい。Spring Security 相当の認証・認可フレームワークの成熟度が低い。既存の Spring 関連知識を活用できない

### Jakarta EE（WildFly / Payara）

- **利点**: Java EE 標準仕様への準拠
- **不採用理由**: アプリケーションサーバーのセットアップ・管理が必要で、開発・デプロイの複雑性が増す。Spring Boot の組み込みサーバー方式と比較して運用負荷が高い

## 影響

### 肯定的な影響

- Spring Boot の自動構成により、プロジェクト初期の立ち上げが迅速になる
- Spring エコシステム全体（MVC、Security、Data JPA）を一貫して利用でき、技術スタックの統一性が保たれる
- 豊富な情報源により、小規模チームでも自律的に問題解決できる

### 否定的な影響

- Spring Boot のバージョンアップに追従する必要がある（年 2 回のメジャーリリース）
- フレームワークの「魔法」（自動構成）により、設定の追跡が困難になる場合がある
- メモリ消費量が Quarkus/Micronaut と比較して大きい（本システムの規模では問題にならない）

## コンプライアンス

- `build.gradle` で Spring Boot のバージョンが 3.x 系であることを確認する
- 新規ライブラリ追加時は Spring Boot の BOM（Bill of Materials）との互換性を確認する
