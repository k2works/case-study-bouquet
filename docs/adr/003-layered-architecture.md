# ADR-003: レイヤード3層アーキテクチャを採用

日付: 2026-03-30

## ステータス

承認済み

## コンテキスト

フレール・メモワール WEB ショップシステムのアーキテクチャパターンを決定する必要がある。本システムは受注管理・在庫推移・発注管理を中核とする業務システムであり、ビジネスルール（BR01-BR07）の正確な実装が求められる。

検討にあたり、以下の制約と要件がある:

- 開発チームは 1〜2 名の小規模構成
- 実装コードは未着手であり、シンプルな構造から段階的に開始したい
- 永続化モデルは単一 RDB（PostgreSQL）であり、CQRS は不要
- Spring Boot / Spring MVC を採用済み（ADR-001）
- ドメインモデルパターンによりビジネスロジックをドメイン層に集約する方針

## 決定

アーキテクチャパターンとして **レイヤード3層アーキテクチャ＋ドメインモデル** を採用する。

構成:

1. **プレゼンテーション層**: Spring MVC Controller、リクエスト/レスポンス DTO、Thymeleaf テンプレート連携
2. **アプリケーション層**: アプリケーションサービスによるユースケース実装、トランザクション境界の管理
3. **インフラ層**: JPA Repository 実装、Spring Boot 設定、Spring Security 設定
4. **ドメインモデル（各層から参照）**: エンティティ、値オブジェクト、集約、ドメインサービス

採用理由:

1. **シンプルな構造**: Controller → Service → Repository の直接的な依存関係により、コードの追跡が容易
2. **Spring Boot との親和性**: Spring Boot / Spring MVC の標準的なパッケージ構成と一致する
3. **段階的な複雑化への対応**: シンプルな構造から開始し、必要に応じてリファクタリングできる
4. **チーム規模への適合**: 1〜2 名の開発者が全レイヤーを把握でき、認知負荷が低い
5. **ドメインモデルの独立性**: ドメインモデルを独立モジュールとして各層から参照し、ビジネスロジックの集約を維持する

## 検討した選択肢

### ヘキサゴナルアーキテクチャ（ポート&アダプター）

- **利点**: ドメインの完全な技術独立、ポート/アダプターによる入れ替え可能性
- **不採用理由**: ポート・アダプターのインターフェース定義が多くなり、本システムの規模（15 画面、10 エンティティ）に対してオーバースペック。永続化先が単一 RDB であり、アダプターの入れ替え需要が低い

### クリーンアーキテクチャ

- **利点**: 依存性逆転の徹底、テスト容易性の最大化
- **不採用理由**: ヘキサゴナルと同様、インターフェース層の増加が小規模チームの開発速度を低下させる。ユースケースごとのクラス分割が細かくなりすぎ、ファイル数が増大する

### Vertical Slice Architecture

- **利点**: 機能単位の凝集性、機能間の独立性
- **不採用理由**: Spring Boot の標準的なパッケージ構成と乖離する。機能間で共有するドメインモデル（商品、単品、得意先等）が多く、Slice 間の重複が発生しやすい。1〜2 名のチームでは機能別分割のメリットが薄い

## 影響

### 肯定的な影響

- Spring Boot の標準的なプロジェクト構成に沿うため、新規メンバーの参加障壁が低い
- レイヤー間の依存方向が明確であり、コードレビューでの指摘基準が明確になる
- ドメインモデルを独立モジュールとすることで、ビジネスルールのユニットテストが外部依存なしで実行できる

### 否定的な影響

- レイヤー間でデータの受け渡し（DTO 変換）が発生し、ボイラープレートコードが増える可能性がある
- 機能追加時に複数レイヤーへの変更が必要となり、変更の局所化が難しい場合がある
- ドメインモデルへの依存が各層に分散するため、ドメインモデルの変更影響が広範囲に及ぶ可能性がある

## コンプライアンス

- パッケージ構成が `presentation` / `application` / `infrastructure` / `domain` に分割されていることを確認する
- 依存方向が上位層 → 下位層であること（逆方向の依存がないこと）をアーキテクチャテストで検証する
- ドメインモデルパッケージが Spring 等のフレームワーク依存を持たないことを確認する
