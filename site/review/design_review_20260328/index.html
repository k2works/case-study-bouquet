    設計成果物レビュー結果¶

    レビュー実施日: 2026-03-28

    レビュー対象¶

    • docs/design/architecture_design.md - アーキテクチャ設計
    • docs/design/data_model.md - データモデル設計
    • docs/design/domain_model.md - ドメインモデル設計
    • docs/design/ui_design.md - UI 設計
    • docs/design/__tests__/design_documents.test.mjs - 設計ドキュメントテスト（114 テスト、全パス）

    総合評価¶

    4 件の設計ドキュメントは高い品質を持ち、要件定義（BR01-BR07）からのトレーサビリティが全設計に確保されている。レイヤード 3 層 + ドメインモデルの選定は要件規模に適合しており、集約設計・ドメインサービスの責務分離も的確。テスト 114 件が再発防止テストを含む形で設計を検証しており、TDD の規律が実践されている。一方で、在庫集約の単品 ID 欠落、受注ステータスの状態遷移不整合、注文確定前確認ステップの欠如の 3 点は実装前に解消が必要。

    改善提案（重要度順）¶

    高（実装前に対応すべき）¶

    # 提案 箇所 指摘元 理由
    1 在庫集約に 単品ID フィールドを追加 domain_model.md 在庫クラス Programmer, Architect, TechWriter データモデルの stocks.item_id と不整合。実装者がフィールドを落とすリスク
    2 受注ステータス「届け日変更済み」の扱いを設計間で統一 domain_model.md, data_model.md Architect, TechWriter, Programmer 要件定義の状態遷移とドメインモデルのメソッド定義に不整合
    3 テストの正規表現偽陽性対処（/orders/ が purchase_orders にマッチ） design_documents.test.mjs Tester テストの信頼性に影響
    4 ビジネスルール BR01-BR07 の個別検証テスト追加 design_documents.test.mjs Tester BR01, BR03, BR04, BR05 がテスト未カバー
    5 ステータス定義値の網羅性テスト（6 値・3 値・2 値の個別存在確認） design_documents.test.mjs Tester 現在はパターンマッチのみで値の存在を検証していない

    中（対応推奨）¶

    # 提案 箇所 指摘元 理由
    6 orders.total_amount の導出方法を明記 data_model.md Programmer BR01 により products.price と同値のはずだが未記述
    7 アプリケーション層→インフラ層の直接依存に対する方針明記（DIP 段階的導入の ADR） architecture_design.md Architect Service が Repository 実装に直接依存。テスタビリティに影響
    8 UI 設計に入力バリデーションルール・エラーメッセージ仕様を追加 ui_design.md TechWriter BR07 の届け日制約等がフロントエンドに未反映
    9 管理者認証の設計方針を明確化 architecture_design.md, ui_design.md Architect, TechWriter 管理画面へのアクセス制御方針が未定義
    10 テストのファイル読み込みキャッシュ化（describe スコープで 1 回のみ） design_documents.test.mjs Programmer, Tester 114 テストで同一ファイルを数十回再読み込み
    11 注文確定前の確認ステップ追加（入力→確認→確定の 3 段階） ui_design.md User 誤注文防止。花束は届け日・届け先を間違えると取り返しがつかない
    12 在庫推移画面にアラート機能（品質期限切れ間近・不足単品の一覧表示） ui_design.md User 単品を 1 つずつ確認する構成では確認漏れが発生
    13 発注管理画面に在庫推移の参照情報を表示 ui_design.md User 在庫推移→発注の画面遷移で情報が断絶
    14 データモデルのカラム型・制約テスト追加（UNIQUE, NOT NULL 等） design_documents.test.mjs Tester 設計の意図を保証するテストが不足
    15 得意先集約と届け先集約の関連方針を明記（独立集約として得意先 ID を FK 保持） domain_model.md TechWriter 実装者が迷う可能性
    16 orders テーブルの product_id に BR01 制約注釈を追記 data_model.md TechWriter 1 受注=1 商品が意図的な設計であることを明示

    低（改善の余地あり）¶

    # 提案 箇所 指摘元 理由
    17 テスト命名の日英混在方針の明確化 design_documents.test.mjs Programmer describe は日本語、it は英語
    18 届け先集約に共通値オブジェクト（名前、住所、電話番号）の参照を検討 domain_model.md Programmer 得意先と届け先で同概念を別定義
    19 出荷日判定サービスの必要性再検討（届け日 VO のメソッドで十分か） domain_model.md Architect 単純計算にドメインサービスは過剰の可能性
    20 product_compositions テーブルに created_at/updated_at 追加 data_model.md Architect 他テーブルにはあるが本テーブルのみ欠落
    21 管理者ログイン画面の URL パス明記 ui_design.md TechWriter 顧客向けと共用か別途 /admin/login か不明
    22 アーキテクチャ概要図の config→controller 矢印方向の見直し architecture_design.md TechWriter 呼び出し方向と設定適用方向の混在

    矛盾事項¶

    # 視点 A 視点 B 論点 推奨判断
    1 Architect: Repository インターフェースをドメイン層に配置し DIP 適用すべき Programmer: シンプルな構造から開始する方針に合致 アプリケーション層→インフラ層の依存方向 初期は直接依存で開始し、テスタビリティが課題になった時点で DIP 導入。ADR に記録
    2 Architect: 出荷日判定サービスは不要（届け日 VO で十分） Programmer: 将来の複雑化に備えてサービスとして分離は合理的 出荷日判定の責務配置 現状維持（サービスとして保持）。祝日考慮等の拡張余地を確保

    エージェント別フィードバック詳細¶

    xp-programmer（高: 0 / 中: 3 / 低: 2） **評価**: TDD サイクル遵守、SOLID 原則、シンプルな設計、要件整合いずれも良好。 **主な指摘**: - 【中】テストのファイル読み込みキャッシュ化 - 【中】在庫集約に単品 ID 欠落 - 【中】orders.total_amount の導出方法未定義 - 【低】テスト命名の日英混在 - 【低】届け先集約の値オブジェクト不足 **良い点**: 非正規化の明示的文書化、集約境界の適切な設計、再発防止テストアプローチ、ユビキタス言語の一貫性。
    xp-tester（高: 3 / 中: 2 / 低: 2） **評価**: テストカバレッジ B+、テスト信頼性 B、境界値 C+、保守性 B+。 **主な指摘**: - 【高】正規表現の偽陽性リスク（`/orders/` が `purchase_orders` にマッチ） - 【高】BR01-BR07 の個別検証テスト不足 - 【高】ステータス定義値の網羅性テスト不足 - 【中】データモデルのカラム型・制約テスト追加 - 【中】ファイル読み込みの繰り返し **良い点**: AAA パターン徹底、再発防止テストの体系化、4 文書間整合性テスト、否定テストの活用。
    xp-architect（高: 2 / 中: 3 / 低: 2） **評価**: アーキテクチャ妥当性良好、設計間整合概ね良好（主要不整合 2 点）、要件整合良好。 **主な指摘**: - 【高】アプリケーション層→インフラ層の直接依存（DIP 検討） - 【高】受注テーブルに数量カラムなし（1 受注=1 個の制約を明文化すべき） - 【中】在庫集約に item_id 欠落 - 【中】受注ステータスの状態遷移不整合 - 【中】管理者認証の設計不足 **懸念**: BR07 暫定値の未確定、引当の永続化方法未定義、在庫推移計算のパフォーマンス。
    xp-technical-writer（高: 2 / 中: 4 / 低: 2） **評価**: ドキュメント品質は高い。トレーサビリティ・ユビキタス言語表・選定理由が充実。 **主な指摘**: - 【高】受注ステータス「届け日変更済み」の設計間不整合 - 【高】在庫集約に単品 ID フィールド欠落 - 【中】レイヤー依存方向と DIP の関係が不明確 - 【中】UI 設計に入力バリデーション仕様未記載 - 【中】orders.product_id に BR01 制約注釈を追記すべき - 【中】得意先・届け先の関連方針を明記すべき **懸念**: BR07 暫定値の波及、UC010 確認メール送信がアーキテクチャに未反映、在庫推移のパフォーマンス。
    xp-user-representative（高: 3 / 中: 4 / 低: 2） **評価**: 業務フローと UI 画面の対応関係は良く整理されており、14 画面構成は MVP として妥当。スタッフの日常作業と顧客の注文完了フローの両面で改善点あり。 **主な指摘**: - 【高】注文確定前の確認ステップが必要（入力→確認→確定の 3 段階。花束は届け日・届け先を間違えると取り返しがつかない） - 【高】在庫推移画面にアラート表示と複数単品同時表示が必要（単品を 1 つずつ確認する構成では確認漏れ必至） - 【高】注文画面で数量（個数）入力欄がない（BR01 の「1 受注=1 商品」制約を UI 上で利用者に説明する必要あり） - 【中】商品一覧画面に商品説明・画像領域がない（花束は視覚的商品。名前と価格だけでは選べない） - 【中】発注管理画面に在庫推移の参照情報を表示（在庫推移→発注の画面遷移で情報断絶） - 【中】注文履歴画面でキャンセル・届け日変更の可否を視覚的に明示（期限超過のボタン非活性化） - 【中】入荷管理画面で品質期限日の自動表示（入荷日 + 品質維持日数） - 【低】会員登録画面に郵便番号フィールド追加 - 【低】管理画面ダッシュボード（将来用テンプレート構成の検討） **懸念**: 業務用語「受注番号」を顧客向け画面では「ご注文番号」に使い分けるべき。BR07 暫定値が画面のガイド表示に未反映。

    対応方針¶

    # 指摘 対応方針 タイミング
    1 在庫集約に単品 ID 追加 修正する 即時
    2 受注ステータス「届け日変更済み」の統一 ADR で判断 設計フェーズ内
    3-5 テストの偽陽性・BR 検証・ステータス値検証 修正する イテレーション 1 内
    6-16 中優先指摘 修正する / 保留 イテレーション 1-2 内
    17-22 低優先指摘 保留 必要時に対応

    結論¶

    設計成果物の品質は高く、要件定義からのトレーサビリティ・TDD テスト・ドキュメント間の整合性が確保されている。重要度「高」5 件のうち、在庫集約の単品 ID 追加とテスト修正は即時対応可能。受注ステータスの扱いは ADR で方針を決定する。設計成果物として開発実装フェーズに進むのに十分な品質水準に達している。
