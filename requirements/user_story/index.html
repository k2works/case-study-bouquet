    ユーザーストーリー - フレール・メモワール WEB ショップシステム¶

    概要¶

    システムユースケースから開発可能な粒度のユーザーストーリーを導出する。各ストーリーにはユースケースとのトレーサビリティおよび受け入れ基準（Given-When-Then）を定義する。

    ビジネスルール¶

    ルール 内容
    BR01 1 受注 = 1 届け先 = 1 商品（花束）
    BR02 出荷日 = 届け日の前日
    BR03 発注判断は人間が行う（システムは判断材料を提供）
    BR04 単品ごとに特定の仕入先が決まっている
    BR05 品質維持日数を考慮した在庫管理
    BR06 日別有効在庫 = 現在庫 + 入荷予定 - 受注引当 - 廃棄予定（入荷日を Day 0 起算、日単位カウント）
    BR07 最短届け日: 注文日+3日、最長届け日: 注文日+30日、注文受付期限: 届け日の3日前（※暫定値）

    ユーザーストーリー一覧¶

    US001: 単品マスタ登録（UC001）¶

    ストーリー: フレール・メモワール スタッフとして、単品（花）のマスタを登録したい。なぜなら、商品（花束）の構成に使用する花材を整備する必要があるからだ。

    受け入れ基準:

    正常系: - Given: スタッフがシステムにログインしている - When: 商品管理画面で単品名「赤バラ」、品質維持日数「5日」、仕入先「A花卉農園」を入力して保存する - Then: 単品マスタが登録され、単品一覧に「赤バラ」が表示される

    異常系: - Given: スタッフがシステムにログインしている - When: 単品名を空欄のまま保存する - Then: エラーメッセージが表示され、単品は登録されない

    境界値: - Given: スタッフがシステムにログインしている - When: 品質維持日数を「0日」で登録する - Then: エラーメッセージが表示され、品質維持日数は1日以上が必要と通知される

    US002: 商品（花束）登録（UC001）¶

    ストーリー: フレール・メモワール スタッフとして、商品（花束）を新規登録したい。なぜなら、得意先に提供する商品ラインナップを拡充する必要があるからだ。

    受け入れ基準:

    正常系: - Given: 単品マスタに「赤バラ」「カスミソウ」が登録されている - When: 商品管理画面で商品名「お祝い花束」、説明「赤バラとカスミソウの花束」、価格「5000円」を入力して保存する - Then: 商品マスタが登録され、商品一覧に「お祝い花束」が表示される

    異常系: - Given: スタッフがシステムにログインしている - When: 価格を「0円」で登録する - Then: エラーメッセージが表示され、商品は登録されない

    境界値: - Given: スタッフがシステムにログインしている - When: 価格を「1円」で登録する - Then: 商品マスタが正しく登録される

    US003: 商品構成設定（UC001）¶

    ストーリー: フレール・メモワール スタッフとして、商品（花束）の構成（単品の組合せ）を設定したい。なぜなら、花束に使用する花材の種類と数量を定義する必要があるからだ。

    受け入れ基準:

    正常系: - Given: 商品「お祝い花束」が登録されており、単品「赤バラ」「カスミソウ」が存在する - When: 商品構成として「赤バラ 5本」「カスミソウ 3本」を設定して保存する - Then: 商品構成が正しく保存され、商品詳細に構成が表示される

    異常系: - Given: 商品「お祝い花束」が登録されている - When: 構成を空（単品 0 件）のまま保存する - Then: エラーメッセージが表示され、最低1つの単品が必要と通知される

    境界値: - Given: 商品「お祝い花束」が登録されている - When: 単品「赤バラ」の数量を「1本」で設定する - Then: 商品構成が正しく保存される

    US004: 商品編集・廃止（UC001）¶

    ストーリー: フレール・メモワール スタッフとして、既存の商品を編集または廃止したい。なぜなら、商品ラインナップを最新の状態に保つ必要があるからだ。

    受け入れ基準:

    正常系: - Given: 商品「お祝い花束」が登録されている - When: 商品名を「特選お祝い花束」に変更して保存する - Then: 商品名が更新され、商品一覧に「特選お祝い花束」が表示される

    正常系（廃止）: - Given: 商品「お祝い花束」が登録されている - When: 商品のステータスを「廃止」に変更する - Then: 商品が廃止され、商品一覧画面（顧客向け）に表示されなくなる

    異常系: - Given: 商品「お祝い花束」に未出荷の受注が存在する - When: 商品のステータスを「廃止」に変更する - Then: 警告メッセージが表示され、未出荷の受注があることが通知される

    US005: WEB ショップからの注文（UC002）¶

    ストーリー: 得意先として、WEB ショップから花束を注文したい。なぜなら、大切な人に届け日を指定して花束を届けたいからだ。

    受け入れ基準:

    正常系: - Given: 商品「お祝い花束（5000円）」が登録されており、得意先がログイン済みである - When: 商品を選択し、届け日「2026-04-10」、届け先「東京都新宿区1-1-1 山田太郎」を指定して注文を確定する - Then: 受注が登録され、1 受注 = 1 届け先 = 1 商品のルールに従い、受注ステータスが「受注済み」となる

    異常系（在庫不足）: - Given: 商品「お祝い花束」の構成に含まれる「赤バラ」の届け日時点の有効在庫が不足している - When: 得意先が注文を確定する - Then: 在庫不足が通知され、届け日の変更が提案される

    境界値（最短届け日）: - Given: 本日が「2026-04-01」で、得意先がログイン済みである - When: 届け日を「2026-04-03」（注文日+2日）に指定して注文を確定する - Then: 最短届け日（注文日+3日）を満たさないため、エラーメッセージが表示される

    境界値（最長届け日）: - Given: 本日が「2026-04-01」で、得意先がログイン済みである - When: 届け日を「2026-05-02」（注文日+31日）に指定して注文を確定する - Then: 最長届け日（注文日+30日）を超えるため、エラーメッセージが表示される

    US006: 注文時のメッセージ添付（UC002）¶

    ストーリー: 得意先として、注文時にメッセージを添付したい。なぜなら、花束と一緒に気持ちを伝えたいからだ。

    受け入れ基準:

    正常系: - Given: 得意先が注文画面で商品・届け日・届け先を入力済みである - When: メッセージ欄に「お誕生日おめでとうございます」と入力して注文を確定する - Then: メッセージが受注に紐付けて保存される

    正常系（メッセージなし）: - Given: 得意先が注文画面で商品・届け日・届け先を入力済みである - When: メッセージ欄を空欄のまま注文を確定する - Then: メッセージなしで受注が正しく登録される

    異常系: - Given: 得意先が注文画面でメッセージ欄に入力している - When: 上限文字数を超えるメッセージを入力する - Then: エラーメッセージが表示され、文字数の上限が通知される

    US007: 在庫推移の確認（UC003）¶

    ストーリー: フレール・メモワール スタッフとして、日別の在庫推移を確認したい。なぜなら、品質維持日数を考慮した適切な発注判断を行いたいからだ。

    受け入れ基準:

    正常系: - Given: 「赤バラ」の現在庫が 100本、品質維持日数 5日、入荷予定 50本（4/5着）、受注引当 30本（4/3届け）がある - When: スタッフが在庫推移画面を開く - Then: 日別有効在庫（= 現在庫 + 入荷予定 - 受注引当 - 廃棄予定）が表示される

    異常系: - Given: 在庫データが未登録の単品が存在する - When: スタッフが在庫推移画面を開く - Then: 該当単品の在庫推移は 0 で表示される

    境界値（品質維持日数超過）: - Given: 「赤バラ」が 4/1 に入荷しており、品質維持日数が 5日 である - When: 4/7（Day 6）の在庫推移を表示する - Then: 該当在庫が廃棄予定として表示される

    US008: 仕入先への発注（UC004）¶

    ストーリー: フレール・メモワール スタッフとして、在庫推移に基づいて仕入先に発注したい。なぜなら、受注に必要な単品を適切なタイミングで調達したいからだ。

    受け入れ基準:

    正常系: - Given: 在庫推移の確認が完了し、「赤バラ」の仕入先が「A花卉農園」に設定されている - When: スタッフが「赤バラ 100本」の発注を確定する - Then: 発注が登録され、「A花卉農園」に対して発注ステータスが「発注済み」となる

    異常系: - Given: 単品「新品種チューリップ」の仕入先が未設定である - When: スタッフが発注を確定する - Then: エラーメッセージが表示され、仕入先の設定を促される

    境界値: - Given: 在庫推移で「赤バラ」の不足数が 0 本と計算されている - When: スタッフが発注数量を「0本」で確定する - Then: エラーメッセージが表示され、1本以上の数量が必要と通知される

    US009: 入荷の記録（UC005）¶

    ストーリー: フレール・メモワール スタッフとして、仕入先から納品された単品の入荷を記録したい。なぜなら、在庫をリアルタイムに把握して品質維持日数を管理したいからだ。

    受け入れ基準:

    正常系: - Given: 「赤バラ 100本」の発注が「発注済み」で存在する - When: スタッフが入荷管理画面で入荷数量「100本」を入力して記録する - Then: 入荷が記録され、在庫が 100本分更新され、発注ステータスが「入荷済み」に変更される

    異常系（数量不一致）: - Given: 「赤バラ 100本」の発注が「発注済み」で存在する - When: スタッフが入荷数量「80本」を入力して記録する - Then: 入荷が記録され、在庫が 80本分更新される（発注数量との差異が記録される）

    境界値: - Given: 「赤バラ 100本」の発注が「発注済み」で存在する - When: スタッフが入荷数量「0本」を入力して記録する - Then: エラーメッセージが表示され、1本以上の入荷数量が必要と通知される

    US010: 出荷処理（UC006）¶

    ストーリー: フレール・メモワール スタッフとして、出荷予定の受注を確認して出荷処理をしたい。なぜなら、届け日に届け先へ確実に花束を届けたいからだ。

    受け入れ基準:

    正常系: - Given: 届け日「2026-04-10」の受注が「受注済み」で存在し、本日が「2026-04-09」（出荷日 = 届け日の前日）である - When: スタッフが出荷管理画面で出荷処理を実行する - Then: 受注ステータスが「出荷済み」に変更され、配送手配情報が記録される

    異常系: - Given: 届け日「2026-04-10」の受注が存在するが、結束が未完了である - When: スタッフが出荷処理を実行しようとする - Then: 結束未完了の警告が表示され、出荷処理が保留される

    境界値: - Given: 本日が「2026-04-10」（届け日当日）である - When: スタッフが出荷管理画面を開く - Then: 届け日「2026-04-10」の受注は出荷予定一覧に表示されない（出荷日 = 届け日の前日のため前日に出荷済みであるべき）

    US011: 届け日の変更（UC007）¶

    ストーリー: 得意先として、注文済みの花束の届け日を変更したい。なぜなら、届け日の都合が変わる場合があるからだ。

    受け入れ基準:

    正常系: - Given: 受注ステータスが「受注済み」の注文（届け日「2026-04-10」）が存在する - When: 得意先が届け日を「2026-04-15」に変更を依頼する - Then: 届け日が変更され、出荷日が「2026-04-14」に再計算される

    異常系（在庫不足）: - Given: 受注ステータスが「受注済み」の注文が存在する - When: 得意先が変更後の届け日を指定するが、その日の在庫確保が不可能である - Then: 変更不可が通知され、別の届け日の検討を促される

    境界値（注文受付期限）: - Given: 本日が「2026-04-08」で、受注ステータスが「受注済み」の注文が存在する - When: 得意先が届け日を「2026-04-10」（本日+2日、受付期限の3日前を満たさない）に変更を依頼する - Then: 注文受付期限を満たさないため、エラーメッセージが表示される

    US012: 届け先のコピー（UC008）¶

    ストーリー: 得意先として、過去の届け先情報をコピーして注文したい。なぜなら、リピート注文時の届け先入力の手間を省きたいからだ。

    受け入れ基準:

    正常系: - Given: 得意先に過去の注文履歴があり、届け先「東京都新宿区1-1-1 山田太郎」が保存されている - When: 得意先が注文画面で「過去の届け先から選択」を選び、該当の届け先を選択する - Then: 「東京都新宿区1-1-1 山田太郎」の情報が注文画面にコピーされる

    異常系: - Given: 得意先に過去の注文履歴がない - When: 得意先が注文画面で「過去の届け先から選択」を選ぶ - Then: 「過去の届け先がありません」と表示され、手動入力が促される

    境界値: - Given: 得意先に過去の届け先が 1 件のみ保存されている - When: 得意先が「過去の届け先から選択」を選ぶ - Then: 1 件の届け先が一覧に表示され、選択可能である

    US013: 得意先情報の管理（UC009）¶

    ストーリー: フレール・メモワール スタッフとして、得意先の情報と注文履歴を管理したい。なぜなら、得意先へのサービス品質を向上させたいからだ。

    受け入れ基準:

    正常系: - Given: スタッフがシステムにログインしており、得意先「鈴木花子」が登録されている - When: スタッフが得意先管理画面で「鈴木花子」の連絡先を更新する - Then: 得意先情報が正しく更新され、注文履歴が参照できる

    正常系（新規登録）: - Given: スタッフがシステムにログインしている - When: スタッフが得意先管理画面で新規得意先「田中一郎」の情報を入力して登録する - Then: 得意先が新規登録され、得意先一覧に「田中一郎」が表示される

    異常系: - Given: スタッフがシステムにログインしている - When: スタッフが得意先の名前を空欄のまま保存する - Then: エラーメッセージが表示され、得意先情報は更新されない

    US014: 会員登録・ログイン（UC010）¶

    ストーリー: 得意先として、WEB ショップに会員登録してログインしたい。なぜなら、注文や届け先コピーなどの機能を利用するために本人確認が必要だからだ。

    受け入れ基準:

    正常系（新規会員登録）: - Given: 得意先が WEB ショップの会員登録画面にアクセスしている - When: メールアドレス「hanako@example.com」、パスワード、名前「鈴木花子」、連絡先を入力して登録する - Then: 確認メールが送信され、確認後に会員登録が完了する

    正常系（ログイン）: - Given: 得意先「hanako@example.com」が会員登録済みである - When: メールアドレスとパスワードを入力してログインする - Then: 認証が成功し、商品一覧画面が表示される

    異常系（メールアドレス重複）: - Given: 「hanako@example.com」が既に登録済みである - When: 同じメールアドレスで会員登録を試みる - Then: エラーメッセージが表示され、ログインまたはパスワード再設定が案内される

    異常系（パスワード不正）: - Given: 得意先「hanako@example.com」が会員登録済みである - When: 誤ったパスワードでログインを試みる - Then: エラーメッセージが表示され、再入力が促される

    US015: 注文キャンセル（UC011）¶

    ストーリー: 得意先として、注文済みの花束をキャンセルしたい。なぜなら、注文が不要になる場合があるからだ。

    受け入れ基準:

    正常系: - Given: 得意先がログイン済みで、受注ステータスが「受注済み」の注文が存在する - When: 得意先が注文履歴画面からキャンセルを申請する - Then: 受注ステータスが「キャンセル」に変更され、在庫引当が解除される

    異常系（出荷準備中以降）: - Given: 得意先がログイン済みで、受注ステータスが「出荷準備中」の注文が存在する - When: 得意先がキャンセルを申請する - Then: キャンセル不可が通知され、連絡先が案内される

    境界値: - Given: 得意先がログイン済みで、受注ステータスが「受注済み」の注文が複数存在する - When: 得意先が 1 件の注文のみキャンセルを申請する - Then: 指定した 1 件のみキャンセルされ、他の注文は「受注済み」のまま維持される
