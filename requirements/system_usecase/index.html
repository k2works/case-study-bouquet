    システムユースケース - フレール・メモワール WEB ショップシステム¶

    概要¶

    ビジネスユースケースをシステム境界の視点で詳細化し、完全形式で記述する。

    システムユースケース一覧¶

    UC番号 ユースケース名 関連BUC 主アクター
    UC001 商品マスタ管理 BUC01: 商品企画 スタッフ
    UC002 WEB 受注 BUC02: WEB 受注 得意先
    UC003 在庫推移 BUC03: 仕入れ・入荷 スタッフ
    UC004 発注管理 BUC03: 仕入れ・入荷 スタッフ
    UC005 入荷管理 BUC03: 仕入れ・入荷 スタッフ
    UC006 出荷管理 BUC05: 出荷・配送 スタッフ

    UC010 会員登録・ログイン BUC02: WEB 受注 得意先
    UC011 注文キャンセル BUC02: WEB 受注 得意先

    UC001:🖥️商品マスタ管理🌊¶

    使用時のコンテキスト: スタッフが商品（花束）のラインナップを管理する際に使用する。新規商品の登録、既存商品の構成変更、商品の廃止を行う。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: フレール・メモワール スタッフ

    利害関係者と利益: - スタッフ: 商品ラインナップを効率的に管理したい - 得意先: 魅力的な商品を選びたい

    事前条件:

    • スタッフがシステムにログイン済みである
    • 単品（花）のマスタが登録されている
    • 仕入先のマスタが登録されている

    最低保証: 商品マスタのデータ整合性が保たれる

    成功保証: 商品（花束）の構成が正しく登録・更新される

    トリガー: スタッフが商品管理画面を開く

    主成功シナリオ:

    1. スタッフが商品管理画面で商品一覧を表示する
    2. スタッフが新規商品を登録する
    3. スタッフが商品名、説明、価格を入力する
    4. スタッフが商品構成（単品の組合せ）を設定する
    5. システムが商品構成を検証する
    6. システムが商品マスタを保存する

    拡張:

    • 2a. 既存商品を編集する場合: スタッフが商品一覧から対象商品を選択し編集する
    • 2b. 商品を廃止する場合: スタッフが対象商品のステータスを廃止に変更する
    • 5a. 商品構成が不正な場合: システムがエラーメッセージを表示し、スタッフが修正する

    技術およびデータのバリエーションリスト:

    • 商品構成: 単品の種類と数量の組合せ

    関連情報:

    • 関連画面: 商品管理画面、商品一覧画面
    • 関連エンティティ: 商品（花束）、商品構成、単品（花）

    UC002:🖥️WEB 受注🌊¶

    使用時のコンテキスト: 得意先が WEB ショップから花束を注文する際に使用する。商品を選択し、届け日・届け先・メッセージを指定して注文を確定する。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: 得意先

    利害関係者と利益: - 得意先: 簡単に花束を注文し、届け日に届けてほしい - スタッフ: 正確な受注情報を受け取りたい

    事前条件:

    • 商品マスタに商品が登録されている
    • 得意先が WEB ショップにアクセスしている

    最低保証: 不完全な注文は保存されない

    成功保証: 受注が正しく登録され、受注ステータスが「受注済み」となる

    トリガー: 得意先が商品一覧画面で商品を選択する

    主成功シナリオ:

    1. 得意先が商品一覧画面から花束を選択する
    2. システムが注文画面を表示する
    3. 得意先が届け日を指定する
    4. 得意先が届け先情報（氏名、住所、電話番号）を入力する
    5. 得意先がメッセージを入力する
    6. 得意先が注文を確定する
    7. システムが在庫確認を行う
    8. システムが受注を登録する（1 受注 = 1 届け先 = 1 商品）
    9. システムが注文確認を表示する

    拡張:

    • 4a. 過去の届け先をコピーする場合: 得意先が届け先コピー（UC008）を利用する
    • 7a. 在庫が確保できない場合: システムが在庫不足を通知し、得意先が届け日を変更する

    技術およびデータのバリエーションリスト:

    • 届け先入力: 新規入力または過去の届け先からコピー

    関連情報:

    • 関連画面: 商品一覧画面、注文画面、届け先選択画面
    • 関連エンティティ: 受注、商品、届け先、得意先

    UC003:🖥️在庫推移🌊¶

    使用時のコンテキスト: スタッフが日別の在庫予定数を確認し、発注判断の材料とする際に使用する。品質維持日数を考慮した在庫推移の可視化を行う。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: フレール・メモワール スタッフ

    利害関係者と利益: - スタッフ: 在庫状況を正確に把握し、適切な発注判断をしたい

    事前条件:

    • スタッフがシステムにログイン済みである
    • 在庫データと受注データが登録されている

    最低保証: 在庫データの整合性が保たれる

    成功保証: 日別の在庫推移が正しく表示される

    トリガー: スタッフが在庫推移画面を開く

    主成功シナリオ:

    1. スタッフが在庫推移画面を開く
    2. システムが単品ごとの在庫推移を計算する
    3. システムが品質維持日数を考慮した日別在庫予定数を表示する
    4. スタッフが在庫推移を確認する
    5. スタッフが発注が必要な単品を特定する

    拡張:

    • 3a. 品質維持日数を超過する在庫がある場合: システムが廃棄予定として警告を表示する
    • 5a. 発注が必要な場合: スタッフが発注管理（UC004）に進む

    技術およびデータのバリエーションリスト:

    • 表示期間: 日単位での在庫推移表示

    関連情報:

    • 関連画面: 在庫推移画面
    • 関連エンティティ: 在庫、単品（花）

    UC004:🖥️発注管理🌊¶

    使用時のコンテキスト: スタッフが在庫推移を確認した上で、仕入先に対して単品（花）の発注を行う際に使用する。発注判断は人間が行い、システムは判断材料を提供する。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: フレール・メモワール スタッフ

    利害関係者と利益: - スタッフ: 適切なタイミングで適切な数量を発注したい - 仕入先: 正確な発注情報を受け取りたい

    事前条件:

    • スタッフがシステムにログイン済みである
    • 単品ごとに特定の仕入先が設定されている
    • 在庫推移の確認が完了している

    最低保証: 発注データの整合性が保たれる

    成功保証: 発注が正しく登録され、発注ステータスが「発注済み」となる

    トリガー: スタッフが発注管理画面を開く

    主成功シナリオ:

    1. スタッフが発注管理画面を開く
    2. システムが在庫推移に基づく発注推奨情報を表示する（判断材料の提供）
    3. スタッフが発注する単品と数量を決定する（人間判断）
    4. システムが単品ごとに特定の仕入先を自動で設定する
    5. スタッフが発注内容を確認する
    6. スタッフが発注を確定する
    7. システムが発注を登録する

    拡張:

    • 3a. スタッフが推奨数量を変更する場合: スタッフが数量を手動で修正する
    • 4a. 仕入先が未設定の場合: システムがエラーメッセージを表示する

    技術およびデータのバリエーションリスト:

    • 発注数量: スタッフが手動で決定（システムは推奨値を提示）

    関連情報:

    • 関連画面: 発注管理画面
    • 関連エンティティ: 単品（花）、仕入先

    UC005:🖥️入荷管理🌊¶

    使用時のコンテキスト: 仕入先から単品（花）が納品された際に、入荷実績を記録し在庫に反映する。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: フレール・メモワール スタッフ

    利害関係者と利益: - スタッフ: 入荷を正確に記録し、在庫をリアルタイムに把握したい

    事前条件:

    • スタッフがシステムにログイン済みである
    • 対応する発注が登録されている

    最低保証: 入荷データの整合性が保たれる

    成功保証: 入荷が正しく記録され、在庫が更新される

    トリガー: 仕入先から単品が納品される

    主成功シナリオ:

    1. スタッフが入荷管理画面を開く
    2. システムが入荷予定（発注済みの一覧）を表示する
    3. スタッフが入荷対象の発注を選択する
    4. スタッフが入荷数量を入力する
    5. システムが入荷を記録する
    6. システムが在庫を更新する
    7. システムが発注ステータスを「入荷済み」に変更する

    拡張:

    • 4a. 入荷数量が発注数量と異なる場合: スタッフが実際の入荷数量を入力する

    技術およびデータのバリエーションリスト:

    • 入荷数量: 発注数量と一致しない場合がある

    関連情報:

    • 関連画面: 入荷管理画面
    • 関連エンティティ: 入荷、在庫、単品（花）

    UC006:🖥️出荷管理🌊¶

    使用時のコンテキスト: 出荷日（届け日の前日）に、出荷予定の受注を確認し出荷処理を行う。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: フレール・メモワール スタッフ

    利害関係者と利益: - スタッフ: 出荷予定を確実に把握し、届け日に届けたい - 得意先: 届け日に花束を受け取りたい

    事前条件:

    • スタッフがシステムにログイン済みである
    • 受注が登録されている
    • 出荷日 = 届け日の前日である

    最低保証: 受注データの整合性が保たれる

    成功保証: 出荷処理が完了し、受注ステータスが「出荷済み」となる

    トリガー: スタッフが出荷管理画面を開く

    主成功シナリオ:

    1. スタッフが出荷管理画面を開く
    2. システムが出荷予定日（届け日の前日）の受注一覧を表示する
    3. スタッフが出荷対象の受注を確認する
    4. スタッフが結束（商品化）の完了を確認する
    5. スタッフが出荷処理を実行する
    6. システムが受注ステータスを「出荷済み」に変更する
    7. システムが配送手配情報を記録する

    拡張:

    • 4a. 結束が未完了の場合: スタッフが結束作業を先に行う
    • 5a. 在庫が不足している場合: システムが在庫不足を通知する

    技術およびデータのバリエーションリスト:

    • 出荷日判定: 出荷日 = 届け日の前日

    関連情報:

    • 関連画面: 出荷管理画面
    • 関連エンティティ: 受注、在庫

    UC007:🖥️届け日変更🌊¶

    使用時のコンテキスト: 得意先が注文済みの花束の届け日を変更したい場合に使用する。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: 得意先

    利害関係者と利益: - 得意先: 届け日を柔軟に変更したい - スタッフ: 変更に伴う在庫・出荷への影響を把握したい

    事前条件:

    • 得意先が注文済みの受注が存在する
    • 受注ステータスが「受注済み」である

    最低保証: 受注データの整合性が保たれる

    成功保証: 届け日が正しく変更され、出荷日が再計算される

    トリガー: 得意先が届け日変更を依頼する

    主成功シナリオ:

    1. 得意先が注文画面から届け日変更を依頼する
    2. システムが変更後の届け日の入力を受け付ける
    3. システムが在庫推移を確認する
    4. システムが変更後の届け日で在庫確保可能か判定する
    5. システムが届け日を変更する
    6. システムが出荷日を再計算する（出荷日 = 届け日の前日）
    7. システムが変更完了を通知する

    拡張:

    • 4a. 在庫確保が不可能な場合: システムが変更不可を通知し、得意先が別の届け日を検討する

    技術およびデータのバリエーションリスト:

    • 届け日: 変更前と変更後の日付

    関連情報:

    • 関連画面: 注文画面
    • 関連エンティティ: 受注

    UC008:🖥️届け先コピー🌊¶

    使用時のコンテキスト: リピーターの得意先が過去の届け先情報を再利用して、注文時の入力の手間を省く際に使用する。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: 得意先

    利害関係者と利益: - 得意先: 届け先の入力を省略し、簡単に注文したい

    事前条件:

    • 得意先に過去の注文履歴がある
    • 過去の届け先情報が保存されている

    最低保証: 既存の届け先データが変更されない

    成功保証: 過去の届け先情報が注文画面にコピーされる

    トリガー: 得意先が注文画面で届け先コピーを選択する

    主成功シナリオ:

    1. 得意先が注文画面で「過去の届け先から選択」を選択する
    2. システムが届け先選択画面を表示する
    3. システムが得意先の過去の届け先一覧を表示する
    4. 得意先が届け先を選択する
    5. システムが選択された届け先情報を注文画面にコピーする

    拡張:

    • 3a. 過去の届け先がない場合: システムが「過去の届け先がありません」と表示し、手動入力を促す

    技術およびデータのバリエーションリスト:

    • 届け先情報: 氏名、住所、電話番号

    関連情報:

    • 関連画面: 注文画面、届け先選択画面
    • 関連エンティティ: 届け先、得意先

    UC009:🖥️得意先管理🌊¶

    使用時のコンテキスト: スタッフが得意先の情報や注文履歴を管理する際に使用する。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: フレール・メモワール スタッフ

    利害関係者と利益: - スタッフ: 得意先情報を正確に管理し、サービス品質を向上させたい

    事前条件:

    • スタッフがシステムにログイン済みである

    最低保証: 得意先データの整合性が保たれる

    成功保証: 得意先情報が正しく登録・更新される

    トリガー: スタッフが得意先管理画面を開く

    主成功シナリオ:

    1. スタッフが得意先管理画面を開く
    2. システムが得意先一覧を表示する
    3. スタッフが得意先を選択する
    4. システムが得意先の詳細情報と注文履歴を表示する
    5. スタッフが得意先情報を確認・更新する
    6. システムが更新内容を保存する

    拡張:

    • 3a. 新規得意先を登録する場合: スタッフが新規登録を選択し、得意先情報を入力する

    技術およびデータのバリエーションリスト:

    • 得意先情報: 名前、連絡先、注文履歴

    関連情報:

    • 関連画面: 得意先管理画面
    • 関連エンティティ: 得意先

    UC010:🖥️会員登録・ログイン🌊¶

    使用時のコンテキスト: 得意先が WEB ショップを利用するために、新規会員登録またはログインを行う際に使用する。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: 得意先

    利害関係者と利益: - 得意先: 簡単に会員登録・ログインして注文を開始したい - スタッフ: 得意先を正確に識別し、注文履歴を紐付けたい

    事前条件:

    • 得意先が WEB ショップにアクセスしている

    最低保証: 認証データの整合性が保たれる

    成功保証: 得意先が認証され、注文可能な状態になる

    トリガー: 得意先がログイン画面または会員登録画面を開く

    主成功シナリオ（新規会員登録）:

    1. 得意先が会員登録画面を開く
    2. 得意先がメールアドレス・パスワード・名前・連絡先を入力する
    3. システムが入力内容を検証する
    4. システムが確認メールを送信する
    5. 得意先が確認メールのリンクをクリックする
    6. システムが会員登録を完了する

    主成功シナリオ（ログイン）:

    1. 得意先がログイン画面を開く
    2. 得意先がメールアドレスとパスワードを入力する
    3. システムが認証を行う
    4. システムがログインを完了し、商品一覧画面を表示する

    拡張:

    • 2a. メールアドレスが既に登録済みの場合: システムがエラーメッセージを表示し、ログインまたはパスワード再設定を案内する
    • 3a（ログイン）. パスワードが不正な場合: システムがエラーメッセージを表示し、再入力を促す
    • 3b（ログイン）. パスワードを忘れた場合: 得意先がパスワード再設定を依頼し、システムが再設定メールを送信する

    技術およびデータのバリエーションリスト:

    • 認証方式: メールアドレス・パスワードによる認証

    関連情報:

    • 関連画面: ログイン画面、会員登録画面
    • 関連エンティティ: 得意先

    UC011:🖥️注文キャンセル🌊¶

    使用時のコンテキスト: 得意先が注文済みの花束のキャンセルを行う際に使用する。

    スコープ: システム

    レベル: ユーザー目的

    主アクター: 得意先

    利害関係者と利益: - 得意先: 不要になった注文をキャンセルしたい - スタッフ: キャンセルに伴う在庫引当の解除を正確に行いたい

    事前条件:

    • 得意先がログイン済みである
    • 受注ステータスが「受注済み」の注文が存在する

    最低保証: 受注データおよび在庫引当データの整合性が保たれる

    成功保証: 受注ステータスが「キャンセル」に変更され、在庫引当が解除される

    トリガー: 得意先が注文履歴画面からキャンセルを選択する

    主成功シナリオ:

    1. 得意先が注文履歴画面から対象の注文を選択する
    2. 得意先がキャンセルを申請する
    3. システムが受注ステータスを確認する
    4. システムがキャンセルを実行する
    5. システムが在庫引当を解除する
    6. システムがキャンセル完了を通知する

    拡張:

    • 3a. 受注ステータスが「出荷準備中」以降の場合: システムがキャンセル不可を通知し、得意先に連絡先を案内する

    技術およびデータのバリエーションリスト:

    • キャンセル可能条件: 受注ステータスが「受注済み」の場合のみ

    関連情報:

    • 関連画面: 注文履歴画面
    • 関連エンティティ: 受注
