# タスク指示書: 花束注文管理システム RDRA 2.0 要件定義書の作成

## 概要

花束注文管理システム「フレール・メモワール WEB ショップシステム」の要件定義書を、RDRA 2.0（リレーションシップ駆動要件分析）に基づいて作成する。4 層（システム価値→システム外部環境→システム境界→システム内部構造）すべてを一括で作成する。

## 対象システム

フラワーショップ「フレール・メモワール」の花束注文管理システム（WEB ショップシステム）

## 作業内容

### 優先度: 高

#### 1. 成果物ファイルの作成

- **作成ファイル**: `docs/requirements/requirements_definition.md`
- テンプレート `docs/template/要件定義.md` の構造をコピーし、以下の 4 層すべてを花束注文管理システムの内容で埋める

#### 2. 層 1: システム価値

- **システムコンテキスト図**: PlantUML で作成。アクターは得意先（個人顧客）、フレール・メモワール（スタッフ）、仕入先の 3 者。WEB ショップシステムを中心に配置
- **要求モデル**: 各アクターの要求を構造化。ビジネスアーキテクチャ分析書のビジネスシナリオ（問題の定義）とインセプションデッキのビジネス目標を入力とする

#### 3. 層 2: システム外部環境

- **ビジネスコンテキスト**: 組織構造（店舗運営/営業・受注/仕入・在庫/配送/IT）と業務の位置づけ。ビジネスアーキテクチャ分析書の組織マップを入力とする
- **ビジネスユースケース**: バリューストリームの 5 ステージ（商品企画、WEB 受注、仕入れ・入荷、結束、出荷・配送）を BUC として整理
- **業務フロー**: 各 BUC のアクティビティ図を PlantUML で作成。ビジネスアーキテクチャ分析書のアクティビティ図を詳細化する
- **利用シーン**: 得意先の注文シーン、スタッフの在庫確認・発注シーンなどを具体的な場面として記述

#### 4. 層 3: システム境界

- **ユースケース複合図**: ビジネスユースケースからシステムユースケースを導出。インセプションデッキのスコープ内フィーチャ（商品マスタ管理、WEB 受注、在庫推移表示、発注管理、入荷管理、出荷管理、届け日変更対応、届け先コピー機能、得意先管理）を UC として定義。各 UC に画面（boundary）、情報（entity）、条件（control）を関連付ける
- **画面・帳票モデル**: 顧客向け画面（商品一覧、注文画面、届け先選択画面）と管理画面（受注一覧、在庫推移、発注管理、出荷管理）を識別
- **イベントモデル**: システム間のイベント（該当する場合）
- **バリエーション・条件**: 商品種別、受注ステータス、在庫ステータスなどのビジネス上重要な区分を定義

#### 5. 層 4: システム内部構造

- **情報モデル**: ビジネスアーキテクチャ分析書の情報マップ（得意先、受注、商品（花束）、商品構成、単品（花）、仕入先、入荷、在庫、届け先）を PlantUML で詳細化
- **状態モデル**: 主要エンティティ（受注、在庫、発注など）の状態遷移図を PlantUML で作成

### 優先度: 中

#### 6. `docs/requirements/index.md` の更新

- 要件定義書の状況を「未作成」から「作成済み」に更新

## ドメイン知識（入力情報の要約）

### アクター

| 種別 | アクター | 役割 |
|------|---------|------|
| ヒューマン | 得意先 | 花束を注文する個人顧客 |
| ヒューマン | フレール・メモワール（スタッフ） | 受注管理、在庫管理、発注判断、結束、出荷 |
| ヒューマン | 仕入先 | 単品（花）を供給するパートナー |
| コンピュータ | WEB ショップシステム | 受注受付、在庫推移表示、届け先管理 |

### スコープ内フィーチャ

| 優先度 | フィーチャ |
|--------|----------|
| 必須 | 商品マスタ管理（花束構成・単品） |
| 必須 | WEB 受注（商品選択・届け日・届け先・メッセージ） |
| 必須 | 在庫推移表示（日別の在庫予定数） |
| 必須 | 仕入先への発注管理 |
| 必須 | 入荷管理 |
| 必須 | 出荷管理（出荷日 = 届け日の前日） |
| 高 | 届け日変更対応 |
| 高 | 届け先コピー機能 |
| 高 | 得意先管理 |

### スコープ外

- 自動発注（人間判断の尊重）
- 法人顧客対応
- 決済処理の実装
- 配送ルート最適化
- マーケティング機能

### 主要情報エンティティ

得意先、受注、商品（花束）、商品構成、単品（花）、仕入先、入荷、在庫、届け先

## 確認方法

- 成果物 `docs/requirements/requirements_definition.md` が存在すること
- RDRA 4 層すべてのセクションが埋まっていること
- PlantUML 図が各セクションに含まれていること
- テンプレートの構造に準拠していること
- ビジネスアーキテクチャ分析書・インセプションデッキとの整合性があること（アクター名、情報名、フィーチャが一致）
