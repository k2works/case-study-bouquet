### Warning 1: UC 複合図に `control` 要素が欠落（スコープ縮小）

**該当ファイル**: `docs/requirements/requirements_definition.md` — UC 複合図セクション全体（行 444〜593）

**何が問題か**: タスク指示書に「各 UC に画面（boundary）、情報（entity）、条件（control）を関連付ける」と明記されている。テンプレート（`docs/template/要件定義.md` 行 406-407）にも `control` 要素が含まれている。しかし、4 つの UC 複合図すべてで `control` 要素が完全に省略されている。

**ビジネス上の影響**: RDRA の UC 複合図における `control` はシステムの動作条件を表す。このシステムでは以下のような条件が UC に紐づくべきだが、未定義：
- WEB 受注 → 在庫確認条件（届け日の在庫が確保可能か）
- 出荷管理 → 出荷日計算条件（出荷日 = 届け日の前日）
- 在庫推移表示 → 品質維持日数チェック条件
- 届け日変更 → 変更可否判定条件

**どう修正すべきか**: 各 UC 複合図の PlantUML に `control` 要素を追加し、バリエーション・条件セクションで定義した条件と関連付ける。例：
```plantuml
control "在庫確認" as c01
UC1 -- c01
```

**ブロッキングとしない理由**: ドキュメントの構造的欠落であり、Policy の REJECT 基準（テスト欠如、バグ、デッドコード等）に直接該当しない。テストも `boundary` と `entity` のみ検証しており、パスしている。

### Warning 2: テストの `\b` 修正が不完全（行 742）

**該当ファイル**: `docs/requirements/__tests__/requirements_definition.test.mjs` 行 742

**何が問題か**: Decisions ログ（決定 #1）で「JavaScript の `\b` は日本語文字間で機能しない」と正しく分析し、行 589 の正規表現を修正した。しかし、行 742 に同じ問題を持つ正規表現が残っている：

```javascript
// 行 742（未修正 — 同じ \b 問題が残存）
const layer3Content = content.split('## システム境界')[1]?.split(/## システム\b(?!境界)/)[0]
```

**実際の影響**: `\b` が日本語文字間でマッチしないため、この `split` は分割に失敗し、`layer3Content` は層 3 から文末（層 4 を含む）までの全内容になる。テストは「層 3 に情報エンティティへの参照がある」ことを検証しているが、実質的に層 4 の情報モデルの内容もチェック対象に含まれており、テストの検証精度が低下している。結果的にテストはパスするが、意図した範囲とは異なる。

**どう修正すべきか**: 行 589 と同じアプローチで修正する：
```javascript
const layer3Content = content.split('## システム境界')[1]?.split(/\n## システム(?!境界)/)[0]
```

### Warning 3: 状態モデルとバリエーション定義の不整合

**該当ファイル**: `docs/requirements/requirements_definition.md`

**何が問題か**: 受注の状態遷移図（行 672-675）に `届け日変更済み` 状態があるが、バリエーション・条件セクションの受注ステータス一覧（行 418-425）にはこの状態が含まれていない。

- 状態モデル: 受注済み → **届け日変更済み** → 受注済み
- バリエーション: 受注済み / 出荷準備中 / 出荷済み / 配送完了 / キャンセル（5 つ）

**どう修正すべきか**: どちらかに統一する：
- (A) バリエーション・条件の受注ステータス表に `届け日変更済み` を追加する
- (B) `届け日変更済み` を独立した状態ではなく `受注済み` 内の一時的な遷移として扱い、状態遷移図から削除する
