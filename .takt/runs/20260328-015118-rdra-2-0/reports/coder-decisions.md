# 決定ログ

## 1. テストの正規表現を修正（RDRA 4 層の順序チェック）
- **背景**: テストの `content.search(/## システム\b(?!価値|外部|境界)/)` が日本語テキストで機能しなかった。JavaScript の `\b` は ASCII の `\w`（`[a-zA-Z0-9_]`）のみで判定するため、日本語文字間では単語境界が検出されず、正規表現が常に `-1` を返していた
- **検討した選択肢**:
  1. `\b` を除去して `(?!...)` のみ使用 → `### システムコンテキスト` 内の部分一致（position 1 で `## システム` がマッチ）でも誤検出する
  2. `\n` で行頭に固定し `\b` を除去（`/\n## システム(?!価値|外部|境界)/`） → 正確に `## システム` の行頭のみマッチし、`### ` 内の部分一致を排除
  3. `^` + `m` フラグ → 動作するが既存テストとの一貫性が低い
- **理由**: 選択肢 2 が最小限の変更で正確に意図通り動作するため採用。`\n` プレフィックスにより `### システムコンテキスト` 等のサブセクション内での誤マッチを完全に防止できる

## 2. ドキュメント内のサブセクションに `#### ` ではなく太字（`**...**`）を使用
- **背景**: テストが `content.split('### ')[0]` や `content.split('## ')[0]` でセクション内容を抽出している。JavaScript の `String.split()` は部分文字列一致のため、`#### ` は `### ` を含み（position 1）、`### ` は `## ` を含む（position 1）。`#### ` サブヘッダーを使うと、テストが意図したセクション全体ではなく最初のサブヘッダーまでの断片のみを取得してしまう
- **検討した選択肢**:
  1. `#### ` サブヘッダーを使う → テストの split ロジックが壊れる（業務フロー・利用シーン・バリエーション・UC 複合図の 4 セクションで失敗）
  2. `**太字**` でサブセクションを区切る → テスト互換で Markdown としても自然
  3. テストの split ロジックを正規表現に変更 → テスト修正が大規模になる
- **理由**: テスト修正を最小限に抑えつつ、Markdown としても自然な構造にするため選択肢 2 を採用。業務フロー・利用シーン・バリエーション・条件・ユースケース複合図・状態モデルの各セクション内で `**太字**` によるサブセクション区切りを使用した
