# 決定ログ

## 1. テストフレームワークに Node.js 組み込みテストランナーを使用
- **背景**: プロジェクトにテストフレームワークが未導入。ドキュメント検証テストのために新たな依存を追加すべきか判断が必要だった
- **検討した選択肢**: Jest / Vitest の導入、Bash スクリプトでの検証、Node.js 組み込みテストランナー（node:test）
- **理由**: Node.js v24 が利用可能で組み込みテストランナーが十分な機能を持つ。ドキュメント検証に外部依存は不要。追加パッケージなしで実行できる

## 2. テストファイルの配置場所
- **背景**: 既存テストのパターンが存在せず、ドキュメント検証テストの配置場所を決める必要があった
- **検討した選択肢**: プロジェクトルートの `__tests__/`、`docs/requirements/__tests__/`
- **理由**: テスト対象（`requirements_definition.md`）に近い場所に配置し、相対パスでの参照を簡潔にするため `docs/requirements/__tests__/` を選択

## 3. テスト粒度の設計
- **背景**: ドキュメント検証テストとして、どの粒度まで検証すべきか判断が必要だった
- **検討した選択肢**: セクション存在確認のみ、内容のキーワード検証まで、PlantUML 構文の完全検証まで
- **理由**: セクション存在 + キーワード検証 + PlantUML 対応チェック + ビジネスルール反映確認の粒度を選択。PlantUML の構文検証はレンダリングツールが必要なため対象外とし、@startuml/@enduml の対応のみ検証する
